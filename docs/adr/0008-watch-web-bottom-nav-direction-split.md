# ADR-0008: 見守りWeb のモバイルナビ — 下タブは固定3、向きでタブ境界を引く

- ステータス: Accepted
- 日付: 2026-07-23

## 背景（Context）

[[見守りWeb]] には長らくブランド表示も共通ヘッダーもなかった（ログイン後はブラウザのタブ以外に「アサトモ」の文字が出ない）。最重要の来訪者は[[招待]]リンクから初めて来た asatomo 未登録の[[見守り者]]候補であり、彼らの信頼はまず「アサトモという名のサービスだ」と認識できること（初認知）から始まる。これを機に、モバイル前提の共通ナビ（上部ブランドヘッダー＋下タブ）を入れる。

ここで CONTEXT.md の役割非対称が設計に直撃する。[[本人]]／[[見守り者]]はアカウント種別ではなく**関係上の役割**で、[[相互見守り]]を標準経路にした結果（[ADR-0004](0004-product-layering-daily-first.md)）、**同一人が本人かつ見守り者**である。さらに CONTEXT.md（[[本人]]・[[近況]]）は「**あなたが見守っている人**」と「**あなたを見守ってくれる人**」を一字違い・無印で同一画面に並べることを繰り返し禁じる。現状コードもこれを守り、見る側の一覧（`WatchDashboard`）と見守ってくれる人の整理（`/connections`）を別ページに分離している。

論点は「**下タブに何を並べ、役割反転をどう捌くか**」である。

## 決定（Decision）

1. **二層のモバイルナビ。** 上部にブランドヘッダー（左「アサトモ」アイコン＋文字／右ハンバーガー）、下部に固定タブ。ログイン前（未ログイン・`/join` 承諾・`/preview`）はヘッダーのみ・下タブなし（3タブは全部ログイン後データが前提）。ただしログイン前でも「アサトモ」ロゴは出す（初見の初認知が最重要な画面だから）。

2. **下タブは全員固定の3つ: わたし（既定・左）／仲間／伝言。役割で出し分けない。** 同一人が両役割ゆえ「どちらのタブセットか」が決まらず、役割で消えるタブは「半分死んだナビ」に見えて初認知・信頼をむしろ削る。片役割しか持たない少数派（純Web 見守り者・完全受動型本人）には、当該タブの中身を**空状態として優雅に見せる**。

3. **向きでタブ境界を引く（本 ADR の核）。**
   - **仲間** = あなたが**見守っている**人（見る側。[[近況]]・[[代理確認]]・エスカレーション）
   - **わたし** = **見られる側の全部**（生存シグナル送信＝様子を伝える／あなたの記録＝透明性／見守ってくれる人の整理／[[招待]]）
   - **伝言** = [[最後の伝言]]

   [[相互見守り]]の相手は、仲間（自分が見守る）と、わたし内の「見守ってくれる人」（相手が見守る）の**両方に現れる**。これは重複ではなく、CONTEXT.md が要求する**向きの分離をタブ境界そのもので体現**したもの。

4. **「仲間」は2層のラベル。** 下タブのラベルは短く温かい「仲間」。だが**開いた先の画面内総称・見出しは「見守っている人」を維持**して向きを明示する（総称に「仲間」を流用すると、一字違いの逆向き＝「見守ってくれる人」と混同するため）。

5. **管理系はハンバーガーへ。** アカウント／利用規約／プライバシーポリシー／ログアウト。最頻でない管理を下タブから追い出す。

6. **ブランドはロゴ「アサトモ」単体**（プラットフォーム識別子「見守りWeb」「Web」はロゴに載せない）。[[見守りWeb]] は PWA 化してインストール可能にする方針だが、識別が要る場面は将来のシグナル源ファミリー名（例:「アサトモ目覚まし」）とアイコンの意匠差で担う。PWA インストール導線はネイティブアプリを持たない人（iPhone [[本人]]・純Web [[見守り者]]）に絞り、Android 本人でホーム画面にアイコンが2つ並ぶ状況を作らない。

7. **URL 規約。** 既定タブは `/me`（ログイン後 `/`→`/me` リダイレクト）。仲間＝`/watch`、伝言＝`/messages` を維持（URL は内部識別子で生活者コピーに出ないため「最後の伝言」への改称に追随させない）。`/` は未ログインのランディング（`/login` と分離）。ランディングの中身は暫定とし、本格化は宣伝フェーズ（MEMORY の「宣伝は最終フェーズ／初見の橋渡しは急がない」方針に従う）。

## 結果（Consequences）

**正**

- 向きが**タブ境界**で分かれ、CONTEXT.md の「一字違いの同居」禁止に**構造的に触れない**。既存の別ページ分離もそのまま活きる。
- タブが全員固定なので役割で変わらない安定したナビになり、本 ADR の動機（初認知・信頼）を損なわない。
- 本人の最頻アクション（生存シグナル送信）が既定タブ「わたし」から最短。「ただの目覚まし時計に見える」本人UX（CONTEXT.md）と整合。
- ロゴ「アサトモ」がどのログイン後画面にも常駐し、初見の未登録者に現在地を与える。

**負 / 要フォロー**

- [[相互見守り]]の相手が仲間・わたしの**2タブに現れる**。CONTEXT.md 的には正しい分離だが、UX的に冗長に映りうる。冗長が実害化するなら代替案の案B（1人1カード双方向）へ育てる余地を残す。
- 「仲間」という向きを持たない温かい語を**画面内総称に流用すると即座に禁を破る**。2層の規律（ラベルは仲間／総称は見守っている人）を実装・レビューで守り続ける必要がある。
- ~~招待CTA（見守り合いに誘う）を「仲間」の空状態・「わたし」のどちらに置くか、重複を許すかは未確定（実装時に決める）。~~ → **下記「実装決定（grill 2026-07-24）」§5 で解決**（わたしに一本化）。
- ランディング（`/`）を暫定で置くため、素で `/` に来た初見者への訴求は当面弱い。招待経由が主導線の現フェーズでは許容。
- 「最後のメッセージ」→「最後の伝言」の改称は CONTEXT.md 反映済みだが、コード（`messages.tsx`・`domain/messages.ts`・`MessageDisclosure.tsx`）と ADR-0002 の表記統一が別タスクとして残る。→ **下記 §8 で本 ADR 実装に同梱**（下タブ「伝言」と本文の不一致を同じ回で閉じる）。

## 実装決定（grill 2026-07-24）

本 ADR を実装するにあたり grill で確定した設計判断。ADR の決定 (1)〜(7) を具体化・補完する（決定そのものの変更ではない）。

1. **共通レイアウトの枠を新設し、ヘッダー・下タブ・認証判定を集約する。** 現状は各ルートが個別にヘッダーを描き、個別に `signed_out` を判定し、個別にログインカードを出している（同じ分岐が7ファイルに散在）。これを URL に寄与しない共通レイアウト（`_app`）1枚に集約し、既存ページ（activity・connections・messages・death）をその下にぶら下げる。**本 ADR は「タブを足す」作業ではなく、ルーティングの土台入れ替えを含む。**

2. **パス→タブは明示マップで持つ（URL はネストしない）。** どのパスがどの下タブに属するかを純関数の表で宣言し、URL 階層とナビ所属を切り離す。`/activity`・`/connections` は URL を変えず「わたし」に属し、`/death/$subjectId` は URL 階層に乗らないまま「仲間」に属する。決定 (7) の「URL は生活者コピーに出ない内部識別子」と整合。

3. **既定タブは全員 `/me` 固定（決定 (7) を維持し、`isSubject` で出し分けない）。** 非相互承諾者（[[招待]]を `mutual:false` で承諾＝見守ってくれる人ゼロ）には既定タブがほぼ空になるが、それでも固定する。既定タブが役割で動くと決定 (2) が避けた不安定さを裏口から復活させ、`mutual` が後に育った日に既定タブが黙って移動する（[ADR-0004](0004-product-layering-daily-first.md) の相互見守り標準経路と不整合）。空の「わたし」は「機能が無い」ではなく「まだ始めていない」として、見守り合いへの勧誘面に使う。

4. **未ログインは各ページのその場カードを廃し、`/login` へリダイレクト。ログイン後は元のページへ戻す。** 決定 (1) の「下タブはログイン後データ前提」と、`/` をランディング化（決定 (7)）した結果生じる「ログインカードの無いページへ『← トップへ』が着地する」行き止まりを、入口一括判定で解消する。ログインUIは現状 `/` に埋め込み済み → `/login` として独立ページに切り出す。

5. **招待CTA（見守り合いに誘う）は「わたし」に一本化。「仲間」の空状態にはボタンを置かない。** [[招待]]の第一の効果は「相手が承諾すると、まず“あなたを見守る”1本ができる＝見守ってくれる人が増える」であり、逆向き（あなたが相手を見守る＝仲間が増える）は相手が承諾画面で「見守り合い」を選んだ時のみ生じる。ゆえに「仲間」空状態にボタンを置くと、押しても仲間が埋まらない守れない約束になる。空状態には期待の説明（「誘った相手が『見守り合い』を選ぶと、ここに現れます」）＋「わたし」への文字リンク1本のみを置く。

6. **「わたし」の並び: 様子を伝える → あなたの記録 → 見守ってくれる人（人数＋警告）→ 見守り合いに誘う。** 見守ってくれる人は**人数を常時表示**し、**開示ライン（[[見守り者]]2人＝不変条件D）を割るときだけ**「このままだと、もしものときに伝言を届けられません」と警告を添える。既定タブは毎日通る場所であり、1人しかいない本人が自分の網の不足に気づく唯一の機会になる。常時は重くせず（本人UXの「目覚まし時計に見える」を守る）、放置すべきでない例外時だけ重くする。見守ってくれる人0人の本人には様子ブロックと人数を出さず、勧誘の空状態カードに差し替える。

7. **ハンバーガーは当面「アカウント」1項目のみ。** 決定 (5) が挙げた利用規約・プライバシーポリシーはページ未整備のため、押して404になる導線は今は出さず、文書が固まってから同じメニューに足す。ログアウトはメニューに直接置かず、既存のアカウント画面（ログアウト・退会が集約済み）に残す（「行き先の一覧」であるメニューに「動作」を混ぜない／二重化を避ける）。

8. **「最後のメッセージ」→「最後の伝言」の表記統一。** 下タブラベル「伝言」と遷移先本文の不一致を閉じる。まず本 ADR 実装（コミット `01996e8`）で**画面表面コピー**を置換（URL `/messages` は据え置き＝CONTEXT.md 通り）。続くフォローアップで**コード内部の表記も統一**した——`messages.tsx`・`domain/messages.ts`・`web/MessageDisclosure.tsx` を筆頭に、コメント/JSDoc、および受取人向け通知メール文面（`notify/notifier.ts`）や `MessageDisclosure` の表面コピー（前回取りこぼし）まで含め、単体「メッセージ」も概念を指す箇所は「伝言」へ揃えた（DB の英語識別子 `legacyMessages`・`messageRecipients` は不変）。**ADR-0002 の題・本文・wikilink も「最後の伝言」に統一済み。** ただし [[最後の伝言]] を参照する他の歴史的 ADR（0001 の題含む・0004・0006・0007）は当時の呼称 `[[最後のメッセージ]]` のままで、CONTEXT.md の現行グロッサリ項目に解決しない。これらの張り替えは別途のドキュメント整備判断として残す。

9. **実装は2段の連続コミットで進める（別ブランチ・別PRには分けない＝ソロ開発・master直方針）。** 第1段=土台（共通枠・`/login` 分離・既存ページのぶら下げ替え・未ログインのリダイレクト統一）で、下タブはまだ点灯させず既存画面が全て従来通り動くことを検証。第2段=見える変化（`/me`・`/watch` 新設と機能移設、下タブ点灯、ハンバーガー、改称）。第1段は「見た目が変わらないのに全ページの土台が変わる」危険な回のため単独で検証可能にし、どちらの段もそれ単独で動く状態で終える。

## 検討した代替案（Alternatives considered）

- **案B（相手1人＝1カードに束ね、カード内で双方向を主語付き表示）**: [[相互見守り]]の実態に忠実で2タブ分裂が起きない。だが CONTEXT.md の主語明示が絶対条件で、一方向つながりも混在するため状態表現が複雑。骨格段階では過剰。決定 (3) の冗長が将来問題化したら移行可能と記録し、今回は却下。
- **役割で下タブを出し分け（本人用／見守り者用）**: 同一人が両役割ゆえ出し分け基準が決まらない。CONTEXT.md の役割定義に反する。却下。
- **タブ境界を「機能」で引く（なふだ方式の踏襲）**: なふだは単一役割ゆえ機能で綺麗に割れるが、アサトモは向きの分離が最優先制約。機能で割ると一字違いの逆向きが同一タブに同居しうる。却下。
- **ロゴに「見守りWeb」を常設**: 初見の用途説明にはなるが、iPhone [[本人]]にとってここは日常の道具でもあり、毎回「見守りWeb」と名乗ると本人UXの道具感を削る。用途説明はログイン前コピーに担わせ、ロゴは「アサトモ」単体。却下。
- **下タブにアカウントを常設／フルの「最後のメッセージ」ラベル**: 前者は最頻でない管理でタブ枠を消費、後者は3等分幅で折り返す。ハンバーガー／短縮ラベル「伝言」を採用し却下。
