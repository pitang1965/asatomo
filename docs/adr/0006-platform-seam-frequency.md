# ADR-0006: プラットフォーム分担 — 頻度で線を引く（毎日はネイティブ、まれで重い判断はWeb）

- ステータス: Accepted
- 日付: 2026-07-20

## 背景（Context）

アサトモは Android ネイティブアプリと Web の2面を持つ。この2面構成は妥協ではなく必然である:

- **Webのみは不可能**: 確実に鳴る目覚まし（exact alarm・ロック画面上の表示・再起動後の復元）は Web / PWA では実現できない。目覚ましは[[生存シグナル]]の心臓であり譲れない。
- **モバイルのみも不可能**: iPhone の[[見守り者]]を切り捨てられない。また[[招待]]の /join（asatomo 未登録者がリンクから承諾する標準経路、[ADR-0004](0004-product-layering-daily-first.md)・[ADR-0005](0005-invitation-flow-trust-model.md)）は Web でしか成立しない。

問題は2面の**境界線の位置**である。[[相互見守り]]を標準経路とした結果（ADR-0004）、同じ人が[[本人]]でも[[見守り者]]でもある。Android の相互見守りユーザーが毎朝アラームを止めるために開くアプリに見守り機能が一切ないと、プロダクトの感情的報酬（「元気が届く」）が最も頻繁に見る画面から抜け落ちる。逆に、見守り者の全機能をネイティブに持ち込むと、死亡確認フローやゼロ知識復号（ブラウザ実装済み、[ADR-0002](0002-final-message-encryption-key-custody.md)）の二重実装で保守が倍になる。

## 決定（Decision）

境界線は**頻度**で引く（ADR-0004 の daily-first と同型の階層化）:

1. **Android アプリ** = [[本人]]の道具（目覚まし・[[生存シグナル]]・[[旅行モード]]）＋ **見守りの一瞥**（見守り対象の[[近況]]リストと[[代理確認]]「無事です」。将来はプッシュ通知の受け口）。
2. **Web** = [[見守り者]]の**完全な面**（iPhone ユーザーはこれで全機能を使える）＋ まれで重いフローの**唯一の実装**（[[死亡認定]]の報告・[[最後のメッセージ]]の開示/復号・/join 承諾）。
3. アプリから重いフローへは**ブラウザで Web を開くリンク**で送る。WebView 埋め込みや暗号処理の Kotlin 移植はしない。セッションは面ごとに独立（Web 側は初回のみログイン）。

## 結果（Consequences）

**正**

- 相互見守りの日常ループ「アラームを止める→ついでに友の元気を見る」が1画面で閉じ、ADR-0004 の主役（日常価値）が最頻画面に宿る。
- 危険で複雑なフロー（死亡確認・ゼロ知識復号）の実装が Web に1本化され、正しさの検証が1箇所で済む。
- iPhone の見守り者は Web だけで完結し、機能差別を受けない。

**負 / 要フォロー**

- [[近況]]の表示はアプリと Web の2箇所になる。ぼかしルール（留守を推測させるシグナルの汎用表現化）の二重実装を避けるため、**表示文の整形はサーバー側で行い、両クライアントは整形済みを表示するだけ**にする。
- ネイティブの「一瞥」の線引きが将来侵食されうる（「ついでにエスカレーション対応も…」）。判断を伴う操作を足すときは本 ADR の頻度基準に照らすこと。

## 検討した代替案（Alternatives considered）

- **アプリを純・本人道具にし、見守りは全部 Web リンク**: 実装は最薄だが、毎日開く画面から「元気が届く」体験が消える。相互見守りを標準経路にした ADR-0004 と不整合。却下。
- **見守り者機能をフルネイティブ実装**: 死亡確認・復号の二重実装で保守倍増。ゼロ知識暗号の移植はバグ＝メッセージ永久喪失に直結。却下。
- **WebView でアプリ内に Web を埋める**: セッション・戻る操作・通知の複雑さの割に、ブラウザで開くのと体験差が小さい。却下。
