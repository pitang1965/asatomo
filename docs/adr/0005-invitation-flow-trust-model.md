# ADR-0005: 見守り招待の導線 — 使い切りリンク・相互デフォルト・承諾で即成立

- ステータス: Accepted
- 日付: 2026-07-19

## 背景（Context）

[[相互見守り]]は [ADR-0004](0004-product-layering-daily-first.md) で**第一級の標準オンボーディング経路**と定めた。ADR-0004 §4 はその招待を「**リンク → ログイン → Web見守り画面（アプリ不要）**・摩擦最小・罪悪感を誘わない」とする。

しかし既存の実装は招待の相手が**既にasatomoユーザーであること**を前提にしていた（`inviteWatcher(watcherUserId)` と、相手ユーザー確定後にしか作れない `connections`）。「まだアカウントの無い相手にリンクを送る」という標準経路を表現できない。

同時に、見守り者は**高信頼の役割**である（本人の[[近況]]が見え、[[死亡認定]]の投票権を持つ）。摩擦最小と、誰でも見守り者化できてしまう危うさの間で線を引く必要がある。

## 決定（Decision）

1. **[[招待]]を第一級のドメイン概念として新設する。** トークン付きリンクを持ち、相手が確定していない「開いた申し出」を表す。承諾されて初めて[[つながり]]（[[相互見守り]]なら双方向2本）へ**昇格**する。`connections` は承諾後に生まれるエッジのまま。

2. **相互見守りをデフォルトの提示にする（見守り合い）＋明示同意。** 承諾で `招待者→承諾者` と `承諾者→招待者` の2本を作る。承諾者には「◯◯を見守り、◯◯にも見守られる」と明示し、**片務（見守るだけ）への引き下げ**を必ず選べる。招待作成時に片務/相互を選ばせる分岐は作らない（招待は常に相互をオファーし、実際の形は承諾者が選ぶ）。

3. **リンクは使い切り＋期限付き。承諾で即成立（招待者の再承認なし）＋即時取消可。** 承諾者が加わったら招待者へ通知する（黙って増えない透明性）。招待者はいつでも取り消せる。

4. **低摩擦を優先できる根拠は、リスクが既存の多層防御で封じられていること。**
   - [[近況]]は意図的に粗く・過去形で、留守推測（「いってきます」等）はぼかす（CONTEXT.md）。誤って入った相手が得る情報は精密な監視にならない。
   - [[死亡認定]]は1人では無力（見守り者2人以上・過半数・猶予・本人取消。[ADR-0001](0001-death-certification-authorization-model.md)）。誤った1人が単独で開示に到達できない。
   - 使い切り＝誤承諾は最大1人、期限で放置リンクは自然失効、取消で即座に無効化。

5. **なふだからの昇格はこの公開招待リンクを入口にする。** アサトモ側に なふだ専用コードは作らず、なふだが同じリンクへ深リンクするだけ（[ADR-0003](0003-nafuda-integration-boundary.md) §1「実行時に なふだに依存しない」を満たす）。

## 結果（Consequences）

**正**

- ADR-0004 の標準経路（アプリ不要・低摩擦・相互見守り）を、既存モデルの矛盾なく実装できる。
- 相互デフォルトが2人問題を双方同時に解き、「友」アイデンティティと文言（見守り合い）に一致する。
- 招待という別エンティティにトークン・寿命・状態を集約でき、`connections`（確定したつながり）と概念が濁らない。

**負 / 要フォロー**

- 「承諾で即成立（再承認なし）」は、リンク漏洩時に意図しない相手が最大1人まで見守り者化しうる。緩和は使い切り・期限・加入通知・即時取消、および §4 の多層防御。想定外の加入が実運用で問題化するなら、招待者の再承認ステップを後から足せる（ただし摩擦増）。
- 承諾者が「見守るだけ」を選ぶと相互が片側だけになり、招待者は開示の2人に届かない場合がある。これは日常価値（ADR-0004）が支える許容範囲。

## 検討した代替案（Alternatives considered）

- **招待者の再承認あり（承諾→招待者が承認して初成立）**: より安全だが往復2回で ADR-0004 の低摩擦に反する。§4 のリスク封じ込めがあるため過剰と判断。却下。
- **再利用リンク（1リンクを多数に配布）**: SNS公開で見知らぬ多数が見守り者化しうる。1:1の高信頼関係と噛み合わない。却下（複数人を招くなら使い切りリンクを複数発行）。
- **片務をデフォルト**: 相互見守りを標準経路とする ADR-0004 §3 に反し、標準経路が「オプトインの手間」に格下げされる。却下。
- **`inviteWatcher(watcherUserId)` を拡張して招待エンティティを作らない**: トークン・期限・状態を `connections` に載せることになり、「相手未確定の申し出」と「確定したつながり」が同居して概念が濁る。却下（既存関数は承諾解決の内部ロジックとして残す）。
