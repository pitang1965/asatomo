# ADR-0007: アカウント削除 — 即時・不可逆・ハード削除、防御は情報つきの摩擦

- ステータス: Accepted
- 日付: 2026-07-21

## 背景（Context）

[[アカウント削除]]を新設するにあたり、これが「設定画面の一機能」ではなく**安全イベント**であることが問題の中心にある。スキーマ上、ユーザーは2つの役割で他人と結ばれる（`connections` の有向エッジ。`subjectUserId`＝[[本人]]、`otherUserId`＝**誰かの[[見守り者]]／[[受取人]]**）。ゆえに削除は、自分が本人として持つ側の全撤去に加えて、**自分が見守り者を務める全つながりの一括[[見守り解除]]**でもある。

[[見守り解除]]（CONTEXT.md）は既に「見守り者が黙って抜けると本人の網が縮む」ことを沈黙させないために非対称通知を設計済みで、この原則を継承する必要がある。同時に、Google Play はアカウント作成を許すアプリに削除導線を要求する。

論点は「削除の**意味論**」と「**可逆性**」だった。特に、誤削除が唯一の見守り者を失わせる危険（→猶予窓が欲しい）と、安全網グラフに曖昧な中間状態を作る危険（→即時にクリーンに畳みたい）が対立する。

## 決定（Decision）

1. **即時・不可逆・ハード削除。ソフト削除（猶予付き復元）は作らない。** 安全網グラフの「半分消えた」曖昧な状態を避ける。猶予窓を設けると、(a) 猶予中もエッジを張ったままなら cron が“去りゆく本人”を監視し続ける、(b) 猶予開始時に畳んで通知するなら復元しても「抜けた」通知は取り消せない——どちらも中途半端。アサトモの「[[沈黙を情報に変える>近況]]」思想は曖昧状態を作らないことと同型。

2. **誤削除への防御は可逆性ではなく「情報つきの摩擦」で与える。** 削除確認画面で、[[見守り解除]]が既に持つ「開示ライン割れをその場で示す」仕組みを**依存者ごとに集計**して見せる（例:「あなたが抜けると Cさんの見守りは0人に」「Dさんは2人→1人で開示ラインを割ります」）。危険度に比例して警告が重くなる。最終確認はチェックボックス1つ＋赤い実行ボタン（名前入力は高齢の単身者に過酷で、重い警告文が十分な摩擦になる）。

3. **見守り者側は一括[[見守り解除]]として畳み、非対称通知を適用。** 網が縮む[[本人]]へ「〇〇さんが見守りをやめました」を通知。本人自身が起点なので、外される側の当事者性に関する見守り解除の規則に従う。

4. **本人側は「利用をやめた」を穏当に通知し、痕跡を破棄。** 自分を見守る[[見守り者]]には[[死亡認定]]/開示と別物の「〇〇さんはアサトモの利用をやめました」を出す（突然の消失を緊急と誤読させない）。[[最後のメッセージ]]・[[受取人]]指定は破棄し、受取人には通知しない。過去の（開示済み含む）認定に残る投票・[[代理確認]]も完全消去する（プライバシー優先／監査の消費者がアプリ内に不在）。

5. **進行中の[[死亡認定]]との衝突を明示的に解く。** (a) 自分に対する認定が進行中に削除操作＝強力な生存証拠→まず本人取消（`cancelled_by_subject`）でキャンセルしアラート済み見守り者へ伝えてから撤去。(b) 他者の認定の投票者として削除するときは既存の**投票取り下げ**経路と同じ（withdraw→定足数再計算、[ADR-0001](0001-death-certification-authorization-model.md) 不変条件B）。

6. **UI 表面でも率直に「アカウントを削除する」と表す（言い換え規則の例外）。** 削除だと明確に認識できることが安全・信頼・ストア要件。思いやりは確認画面が担う。

7. **唯一の実装は[[見守りWeb]]（[ADR-0006](0006-platform-seam-frequency.md)）。** まれで重いフローなので、アバター→`/account`→`/account/delete`（依存者ごとの結果を見せる確認）を Web に1本化。スマホアプリは設定に「アカウントを削除する」を1項目置き、[[見守りWeb]]の `/account` を開くだけ（ネイティブに再実装しない）。

## 結果（Consequences）

**正**

- 安全網グラフが常に「有効」か「不在」の二値で、曖昧な中間状態を持たない。cron・エスカレーションの盲点が生まれない。
- 削除の危険（依存者の網喪失）が、危険度に比例した確認画面で当人に**事前に**可視化される。[[見守り解除]]の既存部品を再利用でき、安全ロジックが1箇所。
- プライバシー第一（ゼロ知識・痕跡を残さない）と一貫。ストアの削除要件も Web リンク誘導で満たす。

**負 / 要フォロー**

- **誤削除は取り消せない。** 唯一の見守り者が誤タップで消えれば依存者の網は即ゼロ。緩和は §2 の情報つき摩擦のみ。実運用で誤削除が問題化するなら、確認の強度を上げる（例: 依存者がいる場合のみ名前入力）余地を残す。
- **過去の開示イベントの監査が欠ける。** 投票者が後から削除すると「なぜ開示されたか」の記録が不完全に。ピア向け安全網として許容と判断（管理者/法的レビューの消費者が不在）。将来監査が要るなら「行は残し identity を剥がす匿名化」へ切替可能。
- Android→Web の削除は**Web 再ログイン**が要りうる（面ごとにセッション独立。ADR-0006）。まれな重フローゆえ許容。
- Web で削除後、その端末のローカル・アラームは**次にアプリを開くまで鳴りうる**（セッションは即失効するがアラームは端末ローカル）。低優先の follow-up。
- スキーマ修正が必要: `connections.otherUserId` の `onDelete: 'set null'` は `connections_party_ck`（user か email のどちらか一方）と衝突し、見守り者を務めるユーザーの削除で制約違反になる。§3 の「見守り者側エッジは行ごと削除」をアプリ層の削除手順で明示的に行う。

## 検討した代替案（Alternatives considered）

- **猶予付きソフト削除（30日復元）**: 誤削除を救えるが、猶予中の安全網が曖昧状態になり、送信済みの離脱通知を取り消せない。§1 の理由で却下。防御は可逆性でなく情報つき摩擦で与える。
- **監査のための匿名化（行を残し identity を剥がす）**: 消費者不在の監査に重装備で altitude が高い。プライバシー第一と不整合。却下（将来必要なら切替可能と記録）。
- **UI を柔らかい言い換えに（「アサトモをやめる」）**: ただの利用停止に読め、データ消去だと伝わらず安全・ストア要件に反する。却下。
- **削除をスマホアプリにネイティブ実装**: 安全の要（依存者ごとの結果を見せる確認）を2面に重複実装。ADR-0006 の「重いフローは Web に1本化」に反する。却下。
