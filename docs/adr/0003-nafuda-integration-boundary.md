# ADR-0003: なふだ連携の境界（疎結合・単独完結・明示同意）

- ステータス: Accepted
- 日付: 2026-07-14

## 背景（Context）

アサトモは既存のデジタル名刺アプリ「なふだ」とスタックを統一し、連携構想を持つ（名刺のつながり → 見守り者招待の導線、同一OAuthプロバイダIDでのアカウント突き合わせ、DBは分離・連携はAPI経由）。

アサトモの核の原則は **疎結合・依存最小・静かに長く動き続ける**。この原則に照らすと3つのリスクがある:

- **結合**: アサトモの招待導線がなふだAPIに依存すると、なふだの障害・仕様変更でアサトモのオンボーディングが壊れる（＝追従地獄の芽）。
- **脆さ / プライバシー**: 「同一プロバイダIDで突き合わせ」は、片方がGoogle・片方がLINEだと一致せず脆い。また黙って突き合わせると本人の同意なくクロスアプリで名寄せされる。
- **信頼の昇格**: なふだの軽い名刺つながりを、生死を託すアサトモのつながりへ黙って引き上げてはならない。

## 決定（Decision）

1. **アサトモは単独で完結する。** 主たる招待は自前（招待リンク → Better Authでログイン → [[つながり]]成立）。**実行時になふだに依存しない。**
2. **なふだ連携は純粋なオプションの便利レイヤー。** 方向は常に「なふだ → アサトモ」: なふだ側から「このつながりをアサトモに招く」操作が、アサトモの公開招待APIを叩く／招待フローへ深リンクする。逆向き（アサトモがなふだに依存）は作らない。
3. **クロスアプリのアカウント突き合わせは黙って自動でやらない。** なふだからの招待を受けた人がアサトモにログインし、同じプロバイダだった場合に「なふだのアカウントと連携しますか？」と**明示同意を求める**経路でのみ行う。
4. **DBは分離**、連携はAPI＋明示同意に限る。
5. **昇格は双方の明示承諾を通す**（[[つながり]]の定義・質問7と整合）。

## 結果（Consequences）

**正**

- なふだの障害・仕様変更でアサトモが壊れない。追従地獄を避けられる。
- 同意なきクロスアプリ名寄せが起きず、プライバシーを守れる。
- 疎結合・依存最小の思想に忠実。

**負 / 要フォロー**

- 招待フローをアサトモ側で作り直すため、なふだの既存グラフを直接再利用できない（重複）。
- クロスアプリの便利さは薄くなる。ユーザーが両アプリで別プロバイダを使うと連携されない。

## 検討した代替案（Alternatives considered）

- **密結合 / DB共有**: 単一障害点・結合を生み、疎結合原則に反する。却下。
- **OAuth IDによる黙示的な自動名寄せ**: 同意なきクロスアプリ名寄せ（プライバシー）・プロバイダ差異で脆い。却下。
