# ADR-0002: 最後の伝言の暗号化と鍵の預け先（ゼロ知識）

- ステータス: Accepted
- 日付: 2026-07-13

## 背景（Context）

[[最後の伝言]]について、4章の3要件は素直には両立しない:

1. 開示成立まで**運営者も読めない**暗号化保存。
2. 本人は生前いつでも**編集・削除**できる。
3. 開示成立後、**受取人が読める**。

矛盾の核: 「運営者も読めない」＝サーバは復号鍵を持てない。だが本人が死ぬと本人はもう鍵を渡せない。にもかかわらず受取人は復号できねばならない。**死んだ本人の代わりに、運営者に見せず、誰が復号鍵を保持するのか。**

本人・運営者どちらが読めるかは信頼モデルの根幹であり、後から変えにくい。ユーザーは「運営者が読めるのは受け入れられない」と明確に判断した。

## 決定（Decision）

**ゼロ知識を要件とする。** 運営者は原理的に平文を復号できない。暗号化・復号はすべて端末側（Web Crypto）で行い、サーバは使用可能な復号鍵を一切保持しない。

**鍵の預け先 = エンベロープ暗号 + 受取人ごとの「合言葉」方式（MVP）**

- 伝言の本文と見出しはランダムなデータ鍵（DEK）で端末側で暗号化する。**DEK は複数の鍵に包む（マルチラップ）**:
  1. **本人自身の鍵**（本人が生前いつでも読み書きするため）。
  2. **[[受取人]]ごとの[[合言葉]]由来鍵**（開示後に各受取人が復号するため）。
- 本人が受取人ごとに合言葉を設定する。合言葉からKDFで導いた鍵で、その受取人向けの DEK ラップを作る。
- サーバは「暗号文」「暗号化した見出し」「本人ラップDEK」「受取人ごとの合言葉ラップDEK」だけを保存し、**どの鍵（合言葉・本人鍵）も保存しない**。
- 開示成立後、受取人が合言葉を入力して端末側で復号する。運営者はどの鍵も知らないため原理的に読めない。
- 合言葉の"預け方"は本人が選ぶ2戦略を推奨プラクティスとして提示する（**どちらもアプリは合言葉を保存しない＝ゼロ知識は不変。紙はシステムの外**）:
  - **(i) 共有知識型**: 「その受取人だけが知っていること」（例:「最初に飼った犬の名前」）を合言葉にする。物理物がなく、生前の事前共有も不要。弱点は受取人が思い出せない可能性。
  - **(ii) 書き残し型**: 紙に書いて自宅の秘密の場所に保管（エンディングノート文化と整合）。本人の記憶・受取人の知識に依存せず死後に確実に残る。**注意**: 見つけた人が全ての伝言を読める物理的単一障害点になる → 「重要書類同様に厳重保管」「合言葉と宛先の対応のみ記し文面は書かない」「在り処は信頼できる1人にだけ伝える」をガイドに添える。
- **本人の生前の読み書き**: 本人は「本人自身の鍵」でDEKを取り出し、いつでも本文・見出しを復号/編集できる（受取人の合言葉を入力する必要はない）。これで「本人はいつでも編集できる」要件をゼロ知識のまま満たす。本人鍵は本人のマスターパスフレーズ等から端末側で導出し、サーバには渡さない。

**受け入れた代償（設計上の必然）**: 合言葉が失われれば、死んだ本人も運営者も復元できず、**伝言は永久に消える**。これはゼロ知識の裏返しであり、仕様として受容する。

**位置づけ**: 開示物は法的な遺言ではない（[ADR-0001](0001-death-certification-authorization-model.md) と整合）。

## 結果（Consequences）

**正**

- 真のゼロ知識。運営者は読めない。
- メールだけの非技術者の受取人でも成立（鍵ペア不要・端末紛失問題なし）。
- サーバの鍵管理が最小限で、ソロ開発・無料枠・放置耐性の思想に合う。

**負 / 要フォロー**

- 合言葉が正しい受取人に届く必要がある（「その人だけが知ること」を合言葉にして緩和）。
- 合言葉紛失＝復旧不能。運営者は助けられない（意図的な仕様だが、UIで強く周知が必要）。
- 本人側に「受取人ごとの合言葉管理」というUX負荷。
- 「全員／グループ」宛の伝言は受取人ごとにDEKを包み直す必要がある。

## 検討した代替案（Alternatives considered）

- **(B) サーバ側の保存時暗号化（KMS）+ 開示ゲート + 監査**: 実装は最軽量だが、運営者が技術的に読めてしまう。ユーザー判断により却下。
- **(A1) 受取人が鍵ペアを持つ**: ゼロ知識だが、非技術者の受取人に秘密鍵管理を強い、端末紛失で読めない。MVPでは却下。受取人がアプリユーザーの場合の追加オプションとして将来検討。
- **(A3) 見守り者クォーラムによる鍵復元（Shamir秘密分散）**: 「クォーラム＝解錠」で概念的に理想だが、復元鍵をサーバに見せないための端末側協調が重い。**v2の将来余地**として保留。

## 将来余地（Future）

- v2: (A3) クォーラム＝解錠 と (A1) アプリユーザー受取人向け鍵ペアの併用を検討する。
