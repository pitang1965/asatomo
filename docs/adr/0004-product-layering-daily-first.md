# ADR-0004: プロダクトの階層 — 日常見守りを主役、開示を深層、相互見守りを標準経路

- ステータス: Accepted
- 日付: 2026-07-14

## 背景（Context）

[ADR-0001](0001-death-certification-authorization-model.md) は[[最後のメッセージ]]開示に見守り者2人以上を要求する。しかし本アプリの主対象は「一人暮らし・オンライン中心の人付き合い」——**現実に信頼できる2人を挙げるのが難しい層**。

放置すると、差別化機能が一番届けたい人にこそロックされる。最悪、「自分には見守ってくれる2人すらいない」と**孤独を強化**し離脱を招く。

一方、コピー「目覚ましを止めるだけで、大切な人に『今日も元気』が伝わる」は*日常の温かさ*を主語にしており、死亡メッセージではない。遺言機能は元々「静かに後半で触れる」方針。

## 決定（Decision）

1. **主役は日常価値**: 生存確認＋[[近況]]の温かさ（「今日も元気」）。**つながり1人でも、相互見守りでも成立**する。これを headline にする。
2. **開示は"深い層"**: [[最後のメッセージ]]開示は見守り者2人以上で開く深層。ロック＝罰ではなく、自然な段階的開放として見せる。遺言機能を静かに扱う方針と整合。
3. **[[相互見守り]]を第一級の標準オンボーディング経路に**: 似た境遇の単身者どうしがゆるく見守り合う。2人問題を解きつつ、アサトモの「友」のアイデンティティそのもの。
4. **招待の摩擦を最小化**（リンク → LINEログイン → Web見守り画面、アプリ不要）。罪悪感を誘わない文言。

## 結果（Consequences）

**正**

- ユーザーは2人集める前から価値を得られる。開示ロックが「罰」から「段階的開放」に変わる。
- 相互見守りがプロダクトのアイデンティティを与え、同時に見守り者調達問題を解く。
- 既存のコピー・ポジショニングと一貫する。

**負 / 要フォロー**

- 相互見守りが「よく知らない相手どうし」になると、見守り者の質（＝死亡を判断し開示を解錠する相手として信頼できるか）が問題になりうる。マッチング・信頼形成に配慮が要る。緩和は「本人が選ぶ」「2人以上のクォーラム」「猶予＋本人取消」。
- 多くのユーザーが見守り者1人に留まると開示機能の利用率は低くなりうる。ただし日常価値がプロダクトを支えるため許容。

## 検討した代替案（Alternatives considered）

- **開示を headline 機能にする**: 主対象に差別化機能がロックされ、離脱・孤独強化のリスク。却下。
- **開示を見守り者1人に緩めてオンボーディングを楽にする**: ADR-0001（単独開錠の禁止）に反する。却下。

## 更新メモ

- 2026-07-20: 上記「背景」で引用した本人向けコピー「大切な人に『今日も元気』が伝わる」は当時のドラフト表現。用語規約の整理により、本人に見せる文脈で[[見守り者]]を指す語は「**見守ってくれる人**」に統一した（「大切な人」は役割を特定しない詩的コピー専用）。本 ADR の決定内容（日常価値を主役・[[相互見守り]]を標準経路）は不変。詳細は [CONTEXT.md](../../CONTEXT.md) の[[見守り者]]／[[本人]]を参照。
